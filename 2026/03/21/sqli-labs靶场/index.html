一、SQL 注入思路介绍

SQL 注入的本质是攻击者将恶意 SQL 代码通过 Payload 插入到 SQL 语句中，而开发者没有对用户输入进行充分的过滤和转义，导致原先的 SQL 语句结构和语义发生改变。

根本原因在于：

  • 开发者过于信任用户输入，没有进行充分验证和转义
  • 使用字符串拼接构造 SQL 语句，而非参数化查询
  • 数据库错误信息被直接返回给前端，为攻击者提供了信息收集的便利

基于上述原因，SQL 注入的防御思路也就清晰了：严格过滤输入、使用参数化查询、不向用户暴露错误信息。

注入的基本思路：

  1. 寻找注入点（用户可控输入的位置）
  2. 尝试 Payload，探测能否改变 SQL 语句的原有结构
  3. 使用闭合符破坏原有语句结构
  4. 拼接、构造恶意 SQL 语句完成非法操作
  5. 利用数据库错误信息进行信息收集
  6. 最终获取数据甚至服务器权限

二、sqli-labs 靶场简介

sqli-labs 是一个专门用于学习和练习 SQL 注入的开源靶场，每个关卡都专注于不同的注入场景和技巧，非常适合手工练习。

三、快速判断注入类型与闭合符（简单环境）

根据传值方式和 Payload 类型，可以分为四种常见场景。

1. GET 数字型

  • ?id=-1 or 1=1 --
    如果正常回显，说明 SQL 语义可能为 false or true 或已被闭合符闭合。

  • ?id=-1 or 1=2 --
    如果无回显，说明 1=2 的变化导致 SQL 语义变为 false or false，注入点属于 GET 数字型。

2. GET 字符型

  • ?id=-1 or 1=1 --
    如果无回显，说明 1=1 并未使 SQL 语义变为 false or true，可排除数字型，注入点可能属于字符型。

  • ?id=-1' or 1=1 --
    如果没有报错（正常回显或无回显），说明闭合符错误，改为双引号继续尝试。
    如果报错提示错误出现在闭合符附近，说明闭合符缺少括号，可添加括号闭合并进一步尝试，例如 ?id=-1') or 1=1 -- 。

  • 持续报错时，继续尝试添加括号闭合，直至正常回显。若正常回显，说明 SQL 语义变为 false or true，闭合符（如 ')）成功闭合了原有语句。

3. POST 数字型

  • -1 or 1=1#
    如果正常回显，表示 SQL 语义为 false or true 或已被闭合符闭合。

  • -1 or 1=2#
    如果无回显，说明 1=2 的变化导致 SQL 语义变为 false or false，注入点属于 POST 数字型。

4. POST 字符型

  • ' or 1=1#
    如果没有报错（正常回显或无回显），说明闭合符错误，改为双引号继续尝试。
    如果报错提示错误出现在闭合符附近，说明缺少括号，可尝试 ') or 1=1# 等。

  • 持续报错时，继续添加括号闭合，直到正常回显，说明闭合符正确。

四、小结

以上是 SQL 注入最基础的手工判断方法。在实际靶机或真实环境中，还可能遇到过滤、编码、WAF 等更复杂的情况，但核心思路不变：破坏原有语句结构 → 构造合法查询 → 获取数据。
