一、SQL 注入思路介绍

SQL 注入的本质是攻击者将恶意 SQL 代码通过 Payload 插入到 SQL 语句中，而开发者没有对用户输入进行充分的过滤和转义，导致原先的 SQL 语句结构和语义发生改变。

根本原因：

  • 开发者过于信任用户输入，没有进行充分验证和转义。
  • 使用字符串拼接构造 SQL 语句，而非参数化查询。
  • 数据库错误信息被直接返回给前端，为信息收集提供便利。

防御思路：严格过滤输入、使用参数化查询、不向用户暴露错误信息。

SQL注入思路：

  1. 寻找可能与数据库产生交互的注入点。
  2. 尝试 Payload 探测注入点类型和闭合符。
  3. 利用数据库错误信息或页面回显进行初步信息收集。
  4. 拼接、构造恶意 SQL 语句完成非法操作。
  5. 最终获取核心数据，甚至尝试获取服务器权限。

二、sqli-labs 靶场简介

sqli-labs 是一个专门用于学习和练习 SQL 注入的开源靶场，每个关卡都专注于不同的注入场景和技巧，非常适合用来熟悉手工注入的各种姿势。

三、快速判断注入类型与闭合符（简单测试环境）

在简单的测试环境中，根据传值方式和 Payload 的响应情况，通常可以通过以下四种场景来进行快速判断：

1. GET 数字型

  • ?id=-1 or 1=1 --+

    如果页面正常回显，说明 SQL 语义可能变成了 false or true（整体为真，或已被闭合符成功闭合）。

  • ?id=-1 or 1=2 --+

    如果页面无回显，说明 1=2 的变化导致 SQL 语义变为了 false or false，由此可判断注入点属于 GET 数字型。

2. GET 字符型

  • ?id=-1 or 1=1 --+

    如果无回显，说明 1=1 并未起作用，SQL 语义没有发生改变，此时可排除数字型，注入点大概率属于字符型。

  • ?id=-1' or 1=1 --+

    加入单引号后，如果没有报错（正常回显或无回显），说明闭合符猜解错误，改为双引号 " 等继续尝试。

    如果报错且提示错误出现在闭合符附近，说明闭合符可能缺少括号，可添加括号闭合并进一步尝试，例如 ?id=-1') or 1=1 --+ 。

  • 测试规律：持续报错时，继续尝试添加各类括号闭合，直至页面正常回显。若正常回显，说明我们构造的闭合符（如 ')）成功闭合了原有语句。

3. POST 数字型

  • -1 or 1=1#

    正常回显，表示 SQL 语义为 false or true 或已被闭合符闭合。

  • -1 or 1=2#

    无回显，说明 1=2 的变化导致 SQL 语义变为 false or false，注入点属于 POST 数字型。

4. POST 字符型

  • ' or 1=1#

    如果没有报错，说明闭合符错误，改为双引号继续尝试。

    如果报错提示错误出现在闭合附近，说明缺少括号，可尝试 ') or 1=1# 等。

  • 持续报错时，继续尝试添加括号闭合，直到正常回显即可确认闭合符。

四、如何快速判断数据库类型

小技巧：默认使用 --+ 或 # 进行注释测试，如果它们失效导致 SQL 语句报错，则说明目标极大概率是非 MySQL 数据库。此时可改用 -- 通用注释符，这样可以快速确认情况并切换注入思路。
