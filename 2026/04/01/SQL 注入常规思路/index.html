一、SQL 注入思路介绍

SQL 注入的本质是攻击者将 SQL 代码通过 Payload 插入到 SQL 语句中，而开发者没有对用户输入进行充分的过滤和转义，导致原先的 SQL 语句 结构和语义 发生 改变。

根本原因：

  • 开发者过于信任用户输入，没有进行充分 验证和转义。
  • 使用字符串拼接构造 SQL 语句，而非 参数化查询。
  • 数据库 错误信息 被直接返回给前端，为渗透提供便利。

防御思路：严格过滤输入、使用参数化查询、避免向用户暴露错误信息。

SQL 注入思路：

  1. 寻找可能与数据库产生交互的 注入点。
  2. 根据页面响应差异反推并绕过 防御机制。
  3. 尝试 Payload 探测 SQL 语句底层结构。
  4. 拼接、构造恶意 SQL 语句完成非法操作。
  5. 最终获取核心数据，甚至实现 提权。

测试规律：

  • 优先构造无效 Payload 尝试 破坏 SQL 结构，依此发现注入点。

  • 再尝试数字型，若无效则转字符型。

  • 字符型测试时，从单引号开始，若报错则尝试添加括号（如 ')、"))），直至页面正常回显。

  • 注意 GET 类型优先使用 --+ 注释，POST 类型优先使用 # 注释。

核心思路：

构造差异化的输入，观察数据库返回的页面表现，反向推断后端 SQL 语句的底层结构。

四、快速判断数据库类型

小技巧：默认 # 作为 POST 传值注释符，如果它失效并导致 SQL 语句报错，则说明目标很可能是非 MySQL 型数据库。此时可改用 -- 注释符，这样可以快速确认数据库类型并切换注入思路。

十一、sqli-labs 靶场实战：Less-9（时间盲注）

1. 判断 SQL 语句底层结构

简单测试发现，无论输入什么 Payload，页面都固定回显 “You are in…”，既无回显位，也无报错信息，且页面内容无任何差异。此时布尔盲注失效，尝试利用 时间盲注（延时注入），根据页面响应时间的差异来推断数据。

首先构造 False or Sleep 布尔条件，通过延时回显判断注入点类型和闭合符：

测试发现，当 Payload 为 ?id=-1' or sleep(5)--+ 时页面出现明显延时，说明闭合符为单引号 '。

2. Payload 换用 and

探测到闭合符后，需要构造条件触发延时。此时选择 and 而非 or，原因在于 or 与 sleep 联用时，WHERE 子句会对每一行独立评估。如果左侧条件为假，右侧 sleep 会对每一行执行一次，导致总延时 = 行数 × 设定延时。对于数据量较大的表，这会造成极长的等待时间，严重影响测试效率。

因此，在已知某条数据存在（如 id=1）的情况下，选择 恒真条件 + and 构型，让延时只在条件为真时触发一次，效率最高。

3. 探测数据库名长度

使用 and 结合 if 函数构造条件，当 if 条件为真时执行 sleep(5)，否则立即返回。

在 Burp Intruder 中使用 Sniper 模式，为参数 n 设置范围 0–9，发送请求：

分析请求结果，当 n=8 时响应时间明显最长，说明该请求触发了 sleep(5)，从而确认数据库名长度为 8。

4. 逐字符猜解数据库名

确定长度后，使用 ascii(mid(database(),i,1)) = x 逐个字符猜解。在 Burp Intruder 中使用 Cluster bomb 模式，设置两个 Payload 集：

  • 位置 i：1–8
  • ASCII 值 x：32–126

发送后，按响应时间降序排列，顶部 8 个请求的响应延时均在 5 秒左右，很有可能是预期的响应包。

依次确定所有字符的 ASCII 值，得到数据库名为 security。

5. 后续爆破

表名、字段名、数据的获取流程与 Less-1 中的联合查询类似，只需将子查询替换为 if + sleep 条件，并用同样的方式逐字符猜解。由于原理相同，此处不再赘述。

核心思路：

当网页将隐藏回显作为防御机制时，根据隐藏内容及回显情况，判断并选择尝试使用报错注入、布尔盲注或时间盲注。
