一、SQL 注入思路介绍

SQL 注入的本质是攻击者将 SQL 代码通过 Payload 插入到 SQL 语句中，而开发者没有对用户输入进行充分的过滤和转义，导致原先的 SQL 语句 结构和语义 发生 改变。

根本原因：

  • 开发者过于信任用户输入，没有进行充分 验证和转义。
  • 使用字符串拼接构造 SQL 语句，而非 参数化查询。
  • 数据库 错误信息 被直接返回给前端，为信息收集提供便利。

防御思路：严格过滤输入、使用参数化查询、不向用户暴露错误信息。

SQL 注入思路：

  1. 寻找可能与数据库产生交互的 注入点。
  2. 尝试 Payload 探测 注入点类型和闭合符。
  3. 利用数据库错误信息或页面回显进行 信息收集。
  4. 拼接、构造恶意 SQL 语句完成非法操作。
  5. 最终获取核心数据，甚至实现 提权。

二、sqli-labs 靶场简介

sqli-labs 是一个专门用于学习和练习 SQL 注入的开源靶场，每个关卡都专注于不同的注入场景和技巧，非常适合用来熟悉手工注入的各种姿势。

三、快速判断注入点类型与闭合符（简单测试环境）

核心思路：构造 False or True 条件，尝试改变 True 的 SQL 语义 判断注入点类型，尝试改变 False 的 SQL 结构 判断闭合符，观察页面表现差异。

根据传值方式分为 GET / POST 两类，每类再根据是否需要引号闭合分为数字型与字符型。

测试规律：

  • 优先构造无效 payload 尝试 破坏SQL 结构，依此发现注入点。

  • 再尝试数字型，若无效则转字符型。

  • 字符型测试时，从单引号开始，若报错则尝试添加括号（如 ')、"))），直至页面正常回显。

  • 注意 GET 类型使用 --+ 注释，POST 类型使用 # 注释。

核心思路：基于差异的逆向推断

这种测试方法的本质是 构造差异化的输入，观察数据库返回的页面表现，反向推断后端 SQL 语句的底层结构。这也是贯穿整个 SQL 注入过程的核心思路。

四、快速判断数据库类型

小技巧：默认使用 --+ 或 # 进行注释测试，如果它们失效导致 SQL 语句报错，则说明目标极大概率是非 MySQL 数据库。此时可改用最标准的 -- 注释符，这样可以快速确认情况并切换注入思路。

4. 经典现象解析

执行上述 Payload 后，页面虽然出现了 You have an error in your SQL syntax 的报错，但这并不表明文件上传失败，这个报错其实是一个假象。其本质是因为 INTO OUTFILE 成功将数据写入了文件，导致数据库没有向前端返回常规的数据结果集（Result Set）。后端 PHP 代码试图去解析一个空的结果集时失败，从而触发了靶场开发者预先写死在代码里的 else 报错分支。

5. 验证成功

黑盒测试中，页面报错的来源往往不明。面对文件上传后的报错，更可靠的做法是主动验证——通过蚁剑（AntSword）等工具直接连接 webshell.php，而非依赖报错信息本身。

核心思路：穿透假象，多维验证

本关揭示了“报错但成功”的现象本质——写入操作早于结果集检查。在实际测试中，绝不能仅依赖单一的页面报错信息做判断，应当主动综合多种方式（如直接访问文件、带外请求 OOB、用蚁剑连接）进行多维验证。

十、sqli-labs 靶场实战：Less-8（布尔盲注）

1. 判断注入点与闭合符

通过初步探测，确认 Less-8 的注入点类型为 GET 字符型，闭合符为 '。

2. 为什么选择布尔盲注

Less-8 的页面表现与 Less-5 类似：无论 SQL 语句是否成功，都只显示固定的“You are in…”或什么都不显示，既无回显位，也无报错信息。此时联合查询和报错注入均无法使用，只能通过布尔盲注——根据条件真假导致的页面差异，逐字符推断数据。

核心思路：优先尝试效率更高的报错注入，当报错注入失效时，再转向布尔盲注。这里根据靶场标题选择盲注，实际测试中应优先尝试 extractvalue 等报错函数。

3. 探测数据库名长度

使用布尔条件 LENGTH(database()) = n，根据SQL语义为false时无回显，为true时有回显的特性，通过页面是否显示“You are in…”来判断长度。

在 Burp Intruder 中使用 Sniper 模式，为参数 n 设置一个范围（如 0–9），发送请求：

通过响应包大小筛选异常值。仅当 n=8 时，响应包大小与其他请求明显不同，这很可能是正常回显，优先检查该响应包内容。

打开响应包的 Render 发现页面是正常回显的。

说明 -1' or length(database())=8 使 SQL 整体语义变为了 True，从而确认数据库名长度为 8。

4. 逐字符猜解数据库名

确定长度为 8 后，使用 ASCII(SUBSTR(database(),i,1)) = x 逐个字符猜解。由于需要同时变化位置 i 和 ASCII 值 x，在 Burp Intruder 中使用 Cluster bomb 模式。

设置两个 Payload 集：

  • 位置 i：1-8
  • ASCII 值 x：32–126（可打印字符范围）

发送后，通过响应包大小筛选正确的组合。在大量请求中，匹配特征快速定位响应。通过对比响应包大小，错误请求集中在 1000 字节以上，而大小为 986 字节的包恰好有 8 个，与数据库名长度一致。这些极有可能是正确的返回包，优先查看其内容，发现页面显示 “You are in…”，确认为正确返回包。

依次确定所有字符的 ASCII 值，得到数据库名为 security 。

5. 后续爆破

表名、字段名、数据的获取流程与 Less-1 中的联合查询类似，只需将子查询替换为布尔条件，并用同样的方式逐字符猜解。由于原理相同，此处不再赘述。

核心思路：特征提取，差异推理

布尔盲注的本质是将数据提取转化为一次次的“真/假”判断。通过精心构造布尔条件，利用页面差异（内容、响应长度等）作为方向，即使在没有回显和报错的环境中，也能精确获取数据。这种“将复杂问题拆解为可判定的二进制问题”的思维，是渗透测试中解决“信息盲区”的核心方法。
