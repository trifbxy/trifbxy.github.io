一、SQL 注入思路介绍

SQL 注入的本质是攻击者将 SQL 代码通过 Payload 插入到 SQL 语句中，而开发者没有对用户输入进行充分的过滤和转义，导致原先的 SQL 语句 结构和语义 发生 改变。

根本原因：

  • 开发者过于信任用户输入，没有进行充分 验证和转义。
  • 使用字符串拼接构造 SQL 语句，而非 参数化查询。
  • 数据库 错误信息 被直接返回给前端，为信息收集提供便利。

防御思路：严格过滤输入、使用参数化查询、不向用户暴露错误信息。

SQL 注入思路：

  1. 寻找可能与数据库产生交互的 注入点。
  2. 尝试 Payload 探测 注入点类型和闭合符。
  3. 利用数据库错误信息或页面回显进行 信息收集。
  4. 拼接、构造恶意 SQL 语句完成非法操作。
  5. 最终获取核心数据，甚至实现 提权。

二、sqli-labs 靶场简介

sqli-labs 是一个专门用于学习和练习 SQL 注入的开源靶场，每个关卡都专注于不同的注入场景和技巧，非常适合用来熟悉手工注入的各种姿势。

测试规律：

  • 优先构造无效 payload 尝试 破坏SQL 结构，依此发现注入点。

  • 再尝试数字型，若无效则转字符型。

  • 字符型测试时，从单引号开始，若报错则尝试添加括号（如 ')、"))），直至页面正常回显。

  • 注意 GET 类型使用 --+ 注释，POST 类型使用 # 注释。

核心思路：

这种测试方法的本质是 构造差异化的输入，观察数据库返回的页面表现，反向推断后端 SQL 语句的底层结构。这也是贯穿整个 SQL 注入过程的核心思路。

四、快速判断数据库类型

小技巧：默认使用 --+ 或 # 进行注释测试，如果它们失效导致 SQL 语句报错，则说明目标极大概率是非 MySQL 数据库。此时可改用最标准的 -- 注释符，这样可以快速确认情况并切换注入思路。

1. 判断注入点与闭合符

通过初步探测，确认 Less-8 的注入点类型为 GET 字符型，闭合符为 '。

2. 为什么选择布尔盲注

Less-8 的页面表现与 Less-5 类似：无论 SQL 语句是否成功，都只显示固定的“You are in…”或什么都不显示，既无回显点，也无报错信息。此时联合查询和报错注入均无法使用，只能通过布尔盲注——根据条件真假导致的页面差异，逐字符推断数据。

核心思路：

优先尝试效率更高的报错注入，当报错注入失效时，再转向布尔盲注。这里根据靶场标题选择盲注，实际测试中应优先尝试 extractvalue 等报错函数。

十一、sqli-labs 靶场实战：Less-9（时间盲注）

1. 判断注入点与闭合符

简单测试发现，无论输入什么 payload，页面都固定回显 “You are in…”，既无回显位，也无报错信息，且页面内容无任何差异。此时布尔盲注也已失效，只能通过时间盲注——根据页面响应时间的差异来推断数据。

首先构造 False or Sleep 条件，通过延时回显判断注入点类型和闭合符：

测试发现，仅当 payload 为 ?id=-1' or sleep(5)--+ 时页面出现明显延时，说明闭合符为单引号 '。

2. 为什么选择 and 而不是 or

在知道闭合符后，需要构造条件触发延时。这里选择 and 而非 or，原因在于 or 与 sleep 联用时，WHERE 子句会对每一行独立评估。如果左侧条件为假，右侧 sleep 会对每一行都执行一次，导致总延时 = 行数 × 设定延时。对于数据量较大的表，这会造成极长的等待时间，严重影响测试效率。

因此，在已知某条数据存在（如 id=1）的情况下，选择 恒真条件 + and 构型，让延时只在条件为真时触发一次，效率最高。

3. 探测数据库名长度

使用 and 结合 if 函数构造条件：当 if 条件为真时执行 sleep(5)，否则立即返回。

在 Burp Intruder 中使用 Sniper 模式，为参数 n 设置范围 0–9，发送请求：

分析请求结果，当 n=8 时响应时间明显最长，说明该请求触发了 sleep(5)，从而确认数据库名长度为 8。

4. 逐字符猜解数据库名

确定长度后，使用 ASCII(SUBSTR(database(),i,1)) = x 逐个字符猜解。在 Burp Intruder 中使用 Cluster bomb 模式，设置两个 Payload 集：

  • 位置 i：1–8
  • ASCII 值 x：32–126

发送后，按响应时间降序排列，顶部 8 个请求的响应延时均在 5 秒左右，这些极有可能是正确的响应包。

依次确定所有字符的 ASCII 值，得到数据库名为 security。

5. 后续爆破

表名、字段名、数据的获取流程与 Less-1 中的联合查询类似，只需将子查询替换为 if + sleep 条件，并用同样的方式逐字符猜解。由于原理相同，此处不再赘述。

核心思路：

在没有任何页面差异的情况下，主动引入“时间”作为新的检测维度。通过构造条件触发 sleep()，将原本不可见的逻辑判断转化为可测量的时间差。

二十一、sqli-labs 靶场实战：Less-24（二次注入）

1. 信息收集：识别业务场景

Less-24 是一个包含 注册、登录、修改密码 功能的网页。初步测试发现，所有输入点均对特殊字符进行了转义（如 ' 被转义为 \'），无法通过常规手法直接注入。这说明开发者已采取基础防御措施。

2. 二次注入原理

二次注入 的核心思路是：将恶意 Payload 存储在数据库中（此时因转义无法生效），当数据库中的数据被二次取出并拼接到新的 SQL 语句时，若该处未做转义，则 Payload 被激活，引发注入。

在本关中，注册功能将用户名存入数据库，修改密码功能从数据库取出用户名拼接到 UPDATE 语句中。如果修改密码处未对用户名进行转义，则存在二次注入。

4. 实战红线提醒

在真实渗透测试中，绝对禁止 使用此类会造成全局破坏的 Payload。一次注入波及所有用户，不仅违反法律法规，也严重违背职业道德。在授权测试中，应优先使用 limit 0,1 等精确控制影响范围的方式，或仅验证漏洞存在即可。

核心思路：

实战中不应只关注输入点的注入，而忽视源代码的闭合。二次注入正是利用“存储时转义、取出时未转义”的逻辑断层，通过构造闭合使恶意数据在二次使用时被激活。防御的关键在于对所有拼接处进行转义或使用参数化查询，而非仅依赖输入点的过滤。
